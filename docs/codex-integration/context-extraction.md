# 上下文提取指南

本文档介绍 codex-delegate skill 的三种上下文提取模式。

## 概述

上下文提取决定了哪些信息会被传递给 Codex。选择合适的模式可以：

- 提高 Codex 的理解准确性
- 减少 token 消耗
- 加快响应速度

## 1. 智能提取模式（默认）

### 工作原理

智能模式自动分析当前上下文，提取最相关的信息：

```
1. 识别当前编辑的文件（如果有）
2. 通过 import/require 分析相关文件
3. 提取最近 3 轮对话摘要
4. 组装结构化 prompt
```

### 提取内容

| 内容类型 | 提取方式 | 示例 |
|---------|---------|------|
| 当前文件 | 从编辑器状态获取 | `src/auth.ts` |
| 相关文件 | 解析 import 语句 | `src/user.ts`, `src/db.ts` |
| 对话摘要 | 提取最近 3 轮 | 用户意图、关键决策 |
| 任务描述 | 从用户消息提取 | "重构 login 函数" |

### 使用场景

- ✅ 日常代码修改
- ✅ 功能开发
- ✅ Bug 修复
- ✅ 代码审查

### 示例

```
用户：用 codex 重构 login 函数

提取的上下文：
- 当前文件：src/auth.ts（包含 login 函数）
- 相关文件：src/user.ts（User 模型）, src/db.ts（数据库操作）
- 对话摘要：用户想要改用 JWT token
- 任务：重构 login 函数
```

## 2. 完整上下文模式

### 工作原理

传递完整的会话历史，包括：

- 所有对话轮次
- 所有工具调用结果
- 完整的文件内容
- 错误信息和日志

### 提取内容

```
完整会话历史
├── 所有用户消息
├── 所有 OMP 响应
├── 所有工具调用结果
├── 所有文件内容
└── 错误和警告信息
```

### 使用场景

- ✅ 复杂的多步骤任务
- ✅ 需要完整历史的问题分析
- ✅ 跨多个文件的系统性问题
- ❌ 简单任务（token 消耗大）

### 示例

```
用户：用 codex（完整上下文）分析为什么系统性能下降

提取的上下文：
- 完整的对话历史（20 轮）
- 所有性能测试结果
- 所有代码修改记录
- 所有错误日志
```

## 3. 手动指定模式

### 工作原理

用户明确指定要包含的文件和范围：

```
--files=file1,file2          # 指定文件
file.ts:10-50                # 指定行范围
```

### 语法

| 语法 | 说明 | 示例 |
|------|------|------|
| `file` | 整个文件 | `src/auth.ts` |
| `file:start-end` | 行范围 | `src/auth.ts:10-50` |
| `file1,file2` | 多个文件 | `src/auth.ts,src/user.ts` |
| 混合 | 文件和范围 | `src/auth.ts:10-50,src/user.ts` |

### 使用场景

- ✅ 大型项目（避免加载无关文件）
- ✅ 敏感文件（只传递必要内容）
- ✅ 精确控制（知道需要什么）
- ✅ 性能优化（减少 token）

### 示例

```
用户：用 codex 优化这些文件：src/auth.ts:10-50, src/user.ts

提取的上下文：
- src/auth.ts 第 10-50 行（login 函数）
- src/user.ts 完整内容
```

## 模式选择指南

### 决策树

```
需要传递上下文给 Codex
│
├─ 任务简单且文件少？
│  └─ 是 → 智能模式（默认）
│
├─ 需要完整历史？
│  └─ 是 → 完整模式
│
└─ 需要精确控制？
   └─ 是 → 手动模式
```

### 对比表

| 特性 | 智能模式 | 完整模式 | 手动模式 |
|------|---------|---------|---------|
| 自动化程度 | 高 | 高 | 低 |
| Token 消耗 | 中 | 高 | 低 |
| 控制精度 | 中 | 低 | 高 |
| 适用场景 | 日常开发 | 复杂分析 | 精确优化 |
| 响应速度 | 快 | 慢 | 快 |

## 最佳实践

### 1. 默认使用智能模式

大多数情况下，智能模式已经足够好。它自动平衡了上下文完整性和 token 效率。

### 2. 复杂任务使用完整模式

当任务涉及多个步骤、多个文件、需要完整历史时，使用完整模式。

### 3. 大项目使用手动模式

在大型项目中，手动指定相关文件可以减少不必要的 token 消耗。

### 4. 混合使用

可以在不同阶段使用不同模式：

```
第一轮：智能模式（快速分析）
第二轮：手动模式（精确优化）
```

## 常见问题

### Q: 智能模式提取的文件不对？

A: 可以切换到手动模式，明确指定需要的文件。

### Q: 完整模式太慢？

A: 考虑使用智能模式或手动模式减少上下文。

### Q: 如何知道提取了哪些内容？

A: skill 要求助手在调用 Codex 前说明本次提取的上下文摘要；具体 UI 展示取决于当前 OMP/技能运行时。
